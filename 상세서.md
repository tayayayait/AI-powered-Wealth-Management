# AI-powered Wealth Management MVP 상세 명세서

| 항목 | 값 |
| --- | --- |
| 문서 목적 | 개발 및 디자인 구현 기준 문서 |
| 적용 범위 | 회원 서비스, 관리자 서비스, AI 챗봇 자산 등록, 포트폴리오 대시보드, 가치평가 시각화, 외부 데이터 연동 |
| 문서 상태 | 구현 기준안 |
| 수정 원칙 | 본 문서의 수치, 명칭, 상태값은 별도 승인 없으면 고정값으로 사용 |

## 1. 문서 목적

- 본 문서는 AI 기반 주식 자산 관리 핀테크 서비스 MVP의 화면, 동작, 데이터, UI 시스템, 예외 처리 기준을 고정한다.
- 디자이너는 본 문서를 기준으로 화면 설계 파일을 작성한다.
- 개발자는 본 문서를 기준으로 라우팅, API, 상태값, 컴포넌트, 반응형, 접근성, 관리자 기능을 구현한다.
- 본 문서에 없는 항목은 임의 확장하지 않는다. 필요한 경우 변경 요청 항목으로 분리한다.

## 2. 제품 범위와 전제

### 2.1 목표

- 사용자가 자연어 채팅으로 보유 자산을 입력하면 시스템이 종목, 수량, 평균단가를 해석하여 포트폴리오에 등록 또는 수정한다.
- 등록된 자산의 현재 가치와 구성 비중을 조회할 수 있어야 한다.
- 클라이언트 고유 가치평가 로직을 엔진 형태로 적용하고 결과를 시각화할 수 있어야 한다.
- 회원/관리자 기능을 통해 서비스 운영이 가능해야 한다.

### 2.2 MVP 범위 포함

- 이메일 기반 회원가입, 로그인, 로그아웃, 마이페이지
- 회원 포트폴리오 대시보드
- 보유 자산 목록/상세/수정/삭제
- OpenAI 기반 자연어 자산 명령 해석
- Yahoo Finance 계열 시세 조회 연동
- 가치평가 결과 시각화
- 관리자 대시보드, 회원 관리, 데이터 동기화 상태 확인, 가치평가 규칙 버전 관리

### 2.3 MVP 범위 제외

- 실제 주문/매매 실행
- 증권사 계좌 직접 연동
- 실시간 체결 데이터 보장
- 세금 계산, 배당 스케줄, 재무제표 전문 분석
- 소셜 로그인
- 모바일 앱 네이티브 개발

### 2.4 불확실 정보

- 클라이언트의 고유 가치평가 수식 자체는 현재 확실한 정보 없음.
- 따라서 본 문서는 가치평가 엔진의 입력값, 출력값, 버전 관리 구조, UI 표시 규칙만 고정한다.
- Yahoo Finance는 공식 공개 엔터프라이즈 SLA가 명확하지 않으므로, MVP에서는 캐시와 장애 대체 표시를 필수로 둔다.

## 4. 권장 기술 구조

### 4.1 프런트엔드

- 프레임워크: `Next.js` App Router + `React` + `TypeScript`
- 스타일: `Tailwind CSS`
- 상태 관리:
  - 서버 상태: `TanStack Query`
  - 로컬 UI 상태: `Zustand` 또는 React Context
- 차트: `Recharts` 또는 동급 라이브러리
- 폼 검증: `react-hook-form` + `zod`

### 4.2 백엔드

- 런타임: `Node.js`
- 프레임워크: `Express` + `TypeScript`
- 검증: `Zod`
- 인증: `JWT` 또는 세션 쿠키 기반 인증
- 권장 정책: 액세스 토큰 15분, 리프레시 토큰 7일, `httpOnly`, `secure`, `sameSite=lax`

### 4.3 데이터/인프라

- 메인 DB: `PostgreSQL 16`
- 캐시: `Redis` 권장
- 시세 캐시 TTL:
  - 장중 현재가: 60초
  - 장후 현재가: 15분
  - 일봉 차트: 24시간
- 로그/에러 추적: `Sentry`
- 배포 기준:
  - Web: Vercel 또는 동급
  - API: AWS ECS/Fargate, Render, Railway 중 1개
  - DB: 관리형 PostgreSQL
- CI: GitHub Actions

### 4.4 외부 연동

- OpenAI:
  - 사용 목적: 자연어 명령 해석
  - 구현 방식: 구조화 JSON 출력 또는 함수 호출
  - 모델 ID는 코드 하드코딩 금지, 환경 변수로 관리
- Yahoo Finance:
  - 사용 목적: 종목 메타데이터, 시세, 과거 가격, 환율 페어 조회
  - 장애 시 정책: 마지막 캐시값 제공 + `stale` 상태 표시

## 7. AI 챗봇 해석 규격

### 7.1 지원 인텐트

| 인텐트 | 설명 | 예시 |
| --- | --- | --- |
| `ADD_ASSET` | 신규 자산 추가 | `애플 주식 10주 보유` |
| `UPDATE_ASSET` | 기존 자산 수량/단가 수정 | `애플 5주 더 샀어` |
| `REMOVE_ASSET` | 자산 제거 | `테슬라 전량 매도` |
| `QUERY_PORTFOLIO` | 포트폴리오 질의 | `내 포트폴리오 비중 보여줘` |

### 7.2 OpenAI 출력 스키마

```json
{
  "intentType": "ADD_ASSET",
  "confidence": 0.94,
  "requiresConfirmation": true,
  "items": [
    {
      "ticker": "AAPL",
      "assetName": "Apple Inc.",
      "market": "US",
      "quantity": 10,
      "avgPrice": null,
      "currency": "USD"
    }
  ],
  "warnings": []
}
```

### 7.3 확인 규칙

- `confidence < 0.85`이면 자동 확정 금지, 확인 단계 강제
- 종목 코드가 다중 매칭되면 사용자가 선택할 때까지 저장 금지
- 수량 누락 시 저장 금지
- 평균단가 누락은 저장 가능하나 UI에 `미입력` 표시
- 삭제 명령은 항상 확인 모달 1회 추가

### 7.4 챗봇 커밋 규칙

1. 사용자가 메시지 입력
2. 서버가 OpenAI로 파싱 요청
3. 파싱 결과를 확인 카드로 렌더링
4. 사용자가 `확인` 클릭 시 DB 반영
5. DB 반영 후 현재가 재조회
6. 대시보드 캐시 무효화 후 새 값 반영

## 12. 사용자 흐름

### 12.1 회원 가입 흐름

1. 사용자는 `/signup` 진입
2. 이름, 이메일, 비밀번호, 비밀번호 확인 입력
3. 프런트에서 형식 검증
4. 서버 성공 시 자동 로그인
5. 첫 진입 사용자는 `/dashboard`로 이동
6. 포트폴리오가 비어 있으면 챗봇 온보딩 카드 자동 노출

### 12.2 챗봇 자산 등록 흐름

1. 사용자가 챗봇 버튼 클릭
2. 예시 메시지 또는 직접 입력
3. 시스템이 파싱 결과 카드 표시
4. 사용자가 확인 시 저장
5. 대시보드 KPI, 자산 목록, 가치평가 영역 동시 갱신

### 12.3 자산 수정 흐름

1. 자산 목록에서 행 클릭
2. 상세 페이지 또는 편집 드로어 진입
3. 수량, 평균단가 수정
4. 저장 시 즉시 시세 재계산
5. 값 변경 이력은 `admin_audit_logs`에 남기지 않고 사용자 액션 로그 테이블 별도 확장 가능

### 12.4 관리자 가치평가 룰 전환 흐름

1. 관리자가 `/admin/data` 진입
2. 활성 룰 버전 확인
3. 신규 버전을 `Draft`에서 `Active`로 전환
4. 시스템이 재계산 작업 큐 시작
5. 완료 후 모든 자산 평가 결과 갱신

  - 데이터 새로고침 기본 주기 표시 전용
- 섹션 4: 계정 작업
  - `로그아웃`
  - `회원 탈퇴`

회원 탈퇴 규칙:

- 2단계 확인
- 탈퇴 즉시 로그인 세션 폐기
- 포트폴리오 데이터는 30일 soft delete 후 파기

### 13.9 공통 AI 챗봇 패널

- 모든 회원 페이지 우하단 플로팅 버튼 클릭으로 열림
- 기본 안내 문구:
  - `보유 자산을 자연어로 입력하세요. 예: 애플 주식 10주 보유`
- 추천 프롬프트 2개 고정:
  - `애플 주식 10주 보유`
  - `테슬라 3주 추가 매수`
- 메시지 타입:
  - 사용자 메시지
  - 시스템 메시지
  - 확인 카드
  - 오류 메시지

확인 카드 구성:

- 제목: `등록 내용을 확인하세요`
- 항목 리스트:
  - 종목명/티커
  - 수량
  - 평균단가
  - 시장
- 버튼:
  - `확인 후 저장`
  - `수정`
  - `취소`

오류 메시지 유형:

- 종목 미식별
- 수량 누락
- 처리 중 네트워크 오류
- 외부 데이터 조회 실패

### 13.10 관리자 로그인 `/admin/login`

- 회원 로그인과 분리
- 폼 폭 420px
- 제목 `관리자 로그인`
- 관리자 아닌 계정 로그인 시도 시:
  - `관리자 권한이 없는 계정입니다.`

### 13.11 관리자 대시보드 `/admin`

- KPI 카드 4개:
  - 총 회원 수
  - 활성 회원 수
  - 오늘 챗봇 파싱 성공률
  - 외부 데이터 장애 건수
- 2행:
  - 좌측 8컬럼: 일별 가입/활성 추이
  - 우측 4컬럼: 시스템 상태 카드
- 시스템 상태 카드 항목:
  - OpenAI API
  - Yahoo Finance 조회
  - 가치평가 작업 큐
  - DB 연결

### 13.12 회원 관리 `/admin/users`

- 상단 필터:
  - 검색
  - 상태
  - 가입일 기간
- 테이블 컬럼:
  - 회원명
  - 이메일
  - 가입일
  - 최근 로그인
  - 보유 자산 수
  - 상태
  - 액션
- 액션 메뉴:
  - 상세 보기
  - 정지
  - 활성화

정지 규칙:

- 정지 즉시 로그인 차단
- 기존 세션 강제 만료
- 이유 입력 모달 필수

### 13.13 데이터 관리 `/admin/data`

- 섹션 1: 시세 동기화 상태
  - 마지막 성공 시각
  - 현재 실패 건수
  - 강제 재동기화 버튼
- 섹션 2: 가치평가 룰 버전
  - 버전명
  - 상태
  - 적용 시작 시각
  - 설명
- 섹션 3: 시스템 작업 로그
  - 최근 20건

룰 버전 상태:

- `DRAFT`
- `ACTIVE`
- `DEPRECATED`

전환 규칙:

- 동시에 `ACTIVE`는 1개만 허용
- `ACTIVE` 전환 시 확인 모달 문구:
  - `활성 룰을 변경하면 모든 가치평가 결과가 재계산됩니다. 계속하시겠습니까?`

## 14. 반응형 기준

### 14.1 데스크톱 1440px 이상

- 사이드바 고정
- KPI 4열
- 차트 2열 구성 유지

### 14.2 노트북 1024px ~ 1439px

- KPI 2x2
- 상세 페이지 우측 패널 높이 자동
- 자산 목록 상단 필터는 2줄 허용

### 14.3 태블릿 768px ~ 1023px

- 사이드바는 오프캔버스 드로어
- 차트는 세로 스택
- 챗봇 패널 폭 100%, 높이 80vh

### 14.4 모바일 360px ~ 767px

- 회원 기능은 사용 가능해야 하나 우선순위는 조회와 등록
- 테이블은 카드 리스트로 변환하지 않는다. 핵심 컬럼만 남기고 가로 스크롤 허용
- KPI는 1열
- 챗봇은 전체 화면 시트

## 15. 접근성 원칙

- 모든 텍스트 대비는 WCAG AA 기준 4.5:1 이상
- 키보드만으로 전 기능 접근 가능해야 한다
- 포커스 링은 제거 금지
- 아이콘 버튼은 `aria-label` 필수
- 모달/드로어 열림 시 포커스 이동 필수
- 토스트는 `aria-live="polite"`
- 오류 메시지는 입력창과 `aria-describedby` 연결
- 차트는 시각 정보 외에 표 형식 대체 데이터 제공
- 색상만으로 상태를 구분하지 않는다. 배지 텍스트 병행
- `prefers-reduced-motion` 환경에서는 카드 진입 애니메이션 제거

## 18. 구현 우선순위

### 18.1 1차 구현

- 인증
- 회원 대시보드
- 보유 자산 목록/상세
- 챗봇 파싱 및 자산 반영

### 18.2 2차 구현

- 회원 관리
- 룰 버전 관리

## 19. 완료 기준

- 회원은 자연어 입력으로 최소 1개 자산을 등록할 수 있어야 한다.
- 등록 자산은 대시보드와 자산 목록에 3초 이내 반영되어야 한다.
- 외부 시세 실패 시 화면이 비지 않아야 하며 마지막 성공 값과 상태 배지가 보여야 한다.
- 관리자 페이지에서 회원 상태 변경과 가치평가 룰 활성 전환이 가능해야 한다.
- 모든 핵심 액션은 로딩, 성공, 실패 상태를 UI에 표시해야 한다.
